# KubeSpaces Operator — Design

Status: **implemented (Pattern B)**. Go operator (controller-runtime,
kubebuilder-style layout) lives in this directory: `cmd/`, `api/v1alpha1/`,
`internal/controller/`, `internal/provisioner/`. The CRD in `config/crd/` is
now generated by controller-gen from `api/v1alpha1/tenant_types.go` — edit the
Go types and run `make manifests`, not the YAML.

## Purpose

Make tenants Kubernetes-native: a `Tenant` custom resource
(`kubespaces.io/v1alpha1`, graduating to `tenant.kubespaces.io/v1` semantics
once stable) that can be created by the API, by `kubectl apply`, or by GitOps
(Flux/Argo) — and reconciled into a provisioned virtual cluster.

## Two integration patterns

### Pattern A — operator as a bridge (original proposal)

Operator watches `Tenant` objects and **calls the backend API** to provision.

```
kubectl/GitOps ──> Tenant CR ──> operator ──> backend API ──> vCluster
portal/CLI    ──────────────────────────────> backend API ──> vCluster
```

- Pro: zero change to the backend; operator is a thin add-on.
- Con: two write paths into provisioning; the DB and the cluster can disagree
  about what exists; operator needs API credentials; reconcile loops fight
  imperative API calls.

### Pattern B — operator owns provisioning (recommended)

Invert the arrow: the **backend API creates/updates `Tenant` CRs**, and the
operator is the only component that provisions (the `pkg/provisioner` logic
moves into the reconciler).

```
portal/CLI ──> backend API ──┐
kubectl/GitOps ──────────────┼──> Tenant CR ──> operator ──> vCluster
                             │        │
                             └── watches status for the UI
```

- The CR is the **single source of truth** for tenant state; Postgres shrinks
  to users/UI metadata.
- One provisioning path; retries, drift correction and status for free via the
  reconcile loop.
- GitOps-native without any extra work — a `Tenant` in git is a tenant.
- Backend loses its `cluster-admin` requirement (only needs RBAC on
  `tenants.kubespaces.io`); the operator carries the provisioning RBAC.

**Recommendation:** since the backend is being refactored anyway, build for
Pattern B. It's also the standard shape reviewers on r/kubernetes will expect.
The CRD below is identical under both patterns, so this decision doesn't block
the chart or the CRD.

## CRD draft

See [config/crd/kubespaces.io_tenants.yaml](config/crd/kubespaces.io_tenants.yaml).

Key choices (all open for discussion):

- **Group `kubespaces.io`, kind `Tenant`, version `v1alpha1`.** Start at
  v1alpha1 — promising `v1` from day one means conversion webhooks the first
  time the schema changes.
- **Cluster-scoped.** A tenant is a top-level platform object, not something
  that lives inside another team's namespace (same choice Capsule made). The
  operator materializes each tenant into its own namespace
  (`spec.targetNamespace`, default `kubespaces-tenant-<name>`).
- `spec`: owner, display name, resource quota, vCluster options (version,
  values overrides).
- `status`: `phase` (Pending/Provisioning/Ready/Deleting/Failed), conditions,
  `kubeconfigSecretRef` pointing at a Secret in the tenant namespace.

## Scaffolding plan (when we start)

- kubebuilder v4, single controller, Go (shares module conventions with the
  refactored backend).
- Deployed by the umbrella chart (`operator.enabled=true`), CRDs installed via
  chart `crds/` directory once stable.
- E2E: kind + a `Tenant` fixture asserted to reach `Ready` — this becomes part
  of the chart acceptance test.
